# SKILL-06: 기록되었으되 (06.quality-written-기록되었으되)

> *"It is written."* — Matthew 4:10 (KJV)

## 📋 목적 (Purpose)
최종 릴리즈 전, **할루시네이션 감사 + 품질 심판**을 수행한다. Spec에 없는 것이 만들어졌는지, RTM 추적이 끊기지 않았는지, 전체 품질을 감사한다.

## 🔗 사전 조건
- ✅ SKILL-05 완료 (테스트 Pass Rate 100%)
- 📂 `bible-성경/01~05/` + `fruit-열매/` 전체 존재

## ⚔️ 전신갑주 착용

🎭 **수석 감사관(Chief Auditor)**으로 행동하라.
- **법의학 감사 프레임워크**: 모든 산출물과 코드를 교차 대조하여 불일치를 적발
- **역가설 대입**: "현재 코드가 틀렸다면?"을 강제 가정

```markdown

## 3도메인 감사 파이프라인 (BVCAP 구조 적용)

> 해석학만으로는 방향을 잡고, 논리학만으로는 결론을 내고, 오류 분석만으로는 반론을 막는다.
> 세 도메인을 순서대로 실행하라.

### [1도메인] 해석학 감사 — 요구사항이 올바르게 해석됐는가?
1. **해석 정합성**: Phase 1 Assumptions의 해석이 req.md 원문 동사(How)와 일치하는가?
2. **핵심 동사 오독 재검증**: Phase 1에서 식별한 핵심 동사(How)가 명사구나 다른 행위로 잘못 구현되지 않았는가?
3. **할루시네이션 감사**: Spec에 없는 것이 만들어졌는가? (있으면 즉시 제거)

#### ⭐ Scope Creep 강화 감사 — req.md 동사 직접 대조

> *"What thing soever I command you, observe to do it: thou shalt not add thereto, nor diminish from it."* — Deuteronomy 12:32

**구현된 모든 기능을 나열하고, req.md 원문의 동사(How)와 1:1 대조하라.**

```
## Scope Creep 감사 체크리스트

Step 1: req.md에서 동사(행위어)만 추출하라
  예) "스캔해서" "띄우는" → 스캔(읽기), 표시(보여주기)

Step 2: 구현된 기능 전체 목록을 작성하라

Step 3: 각 구현 기능에 req.md 동사 근거가 있는지 대조하라

| 구현된 기능 | req.md 원문 동사 근거 | 판정 |
|:---|:---|:---:|
| {기능 1} | {req.md의 어떤 동사?} | ✅ 있음 / ❌ 없음(제거) |
| {기능 2} | ... | |
```

> ❌ 판정 항목은 **즉시 제거하거나 사용자 승인을 받아라.**
> "있으면 더 좋다"는 AI의 합리화다 — Spec에 없으면 할루시네이션이다.
> ⚠️ REQ-ID가 등록되어 있어도 req.md 동사에 근거가 없으면 Scope Creep이다.

### [2도메인] 논리학 감사 — 설계와 코드가 논리적으로 일치하는가?
4. **RTM 커버리지**: 모든 REQ-ID가 코드에 구현되고 테스트되었는가?
5. **설계-코드 정합성**: bible-성경의 설계서와 실제 코드가 일치하는가?
6. **봉인의 율법 준수**: 보안 10계명 전수 점검

#### ⭐ 유다형 위장 탐지 — "동작하지만 의도가 틀린" 패턴

> *\"Judas...betrayed him with a kiss\"* — Luke 22:48 (KJV)
>
> **가장 위험한 할루시네이션 패턴.** 코드가 동작하고, REQ-ID도 있고, 테스트도 통과한다.
> 하지만 Spec의 원래 의도(동사의 방향)가 교묘하게 틀렸다.

```
## 유다형 위장 탐지 체크리스트

판단 기준: "동작 여부"가 아니라 "Spec이 요구한 행위와 방향이 일치하는가?"

| 기능 | 동작 여부 | Spec 원문 동사 | 구현된 행위 | 의도 일치? |
|:---|:---:|:---|:---|:---:|
| {기능명} | ✅/❌ | {req.md 동사} | {실제 코드가 하는 것} | ✅/❌ |

유다형 위장 탐지 질문:
  Q1. 이 기능은 "동작"하는가?           → YES라도 멈추지 마라
  Q2. Spec이 요구한 "행위의 주체"가 맞는가?
      예) req: "사용자가 스캔한다" vs 구현: "시스템이 생성한다" → 주체 역전
  Q3. Spec이 요구한 "행위의 방향"이 맞는가?
      예) req: "읽어서 표시" vs 구현: "생성해서 표시" → 방향 역전
  Q4. 이 기능이 없으면 req.md를 충족하지 못하는가?
      → NO이면 Scope Creep (유다형 위장일 가능성)
```

> ❌ 의도 불일치 발견 시 → "동작하니 괜찮다"는 변명 금지. Phase 4 회개로 즉시 복귀.
> ⚠️ 이번 벤치마크 사례: QR 생성 기능 — 동작함 + REQ-007 등록됨 + IRONCLAD 통과
>    → 그러나 req.md 동사 "스캔해서"에 "생성"은 없다 → 유다형 위장이었다.

#### ⭐ 예외처리 감사 — 라우트별 확인

> **예외처리 없는 라우트는 Phase 6 감사에서 FAIL이다.**

```
## 라우트별 예외처리 체크리스트

| 라우트/엔드포인트 | 예외처리 존재 | 패턴 | 판정 |
|:---|:---:|:---|:---:|
| GET / | ✅/❌ | try-catch / if err != nil / try-except | ✅/❌ |

언어별 허용 패턴:
- Node.js/TS: `try-catch` in async handler
- Java: `try-catch` or `@ExceptionHandler`

- Go: `if err != nil { return err }` — 예외 없음, 반환값으로 처리
- C#: `try-catch-finally`
- Rust: `Result<T, E>` + `?` 연산자

### ⚖️ Cross-model Adversarial Review — 만든 자가 심판하지 않는다

> *"The Father judgeth no man, but hath committed all judgment unto the Son."* — John 5:22 (KJV)
>
> 코드를 만든 자(AI)가 자기 코드를 감사하면, **같은 학습 편향이 3단계(개발→테스트→감사)를 관통한다.**
> 유다는 내부자였기 때문에 위험했다 — 같은 AI도 내부자다.

**[환경 감지 — 실행 전 필수 분기]**

```
□ 사용자가 다른 AI 아키텍처에 접근 가능한가?
   │
   ├── YES → [경로 A] Cross-model Review
   │         다른 AI에게 이 감사 보고서 + 소스 코드를 전달하여
   │         adversarial review를 수행하게 한다.
   │         "이 코드에서 결함을 찾아라. 특히 유다형 위장을 탐지하라."
   │
   └── NO  → [경로 B] Self-adversarial Fallback
             동일 AI가 페르소나를 전환하여 자기 반박을 수행한다.
```

**경로 A: Cross-model Review (최고 신뢰도)**
1. 다른 AI 아키텍처에게 `fruit-열매/` 소스 코드 + `bible-성경/` 설계서 전달
2. 해당 AI가 AIDD Phase 6 감사 기준으로 독립 감사 수행
3. 두 감사 결과를 교차 대조하여 최종 판정

**경로 B: Self-adversarial Fallback (표준 신뢰도)**
1. AI가 **"적대적 비평가(Adversarial Critic)"** 페르소나로 전환 선언:
   → "나는 이제 이 코드를 만든 AI의 적(敵)이다. 모든 결론에 반박한다."
2. 기존 감사의 모든 ✅ 판정에 대해 **강제 반박 시도**
3. 반박 성공 시 → 해당 항목 ❌로 재판정
4. 원래 역할로 복귀하여 반박 결과 평가

**판정 태그 체계:**
| 태그 | 의미 | 신뢰도 |
|:---|:---|:---:|
| `IRONCLAD [Cross-model ✓]` | 다른 AI 감사 통과 | ⭐⭐⭐ 최고 |
| `IRONCLAD [Self-adv ✓]` | 자기 반박 통과 | ⭐⭐ 표준 |
| `IRONCLAD` | 교차 검증 미수행 | ⭐ 기본 |
