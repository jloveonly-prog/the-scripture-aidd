# SKILL-01: 들음 (01.requirement-hearing-들음)

> *"So then faith cometh by hearing, and hearing by the word of God."* — Romans 10:17 (KJV)

## 📋 목적 (Purpose)
`hearing-들음/req.md`의 원본 요구사항을 듣고, 명세서·유스케이스·RTM을 작성하여 `bible-성경/01/`에 정경화한다.

- ✅ Phase 0 완료 (bootstrap-creed-신앙고백.md 읽기, "나는 죄인이다" 선언)
- 📂 `hearing-들음/req.md`에 요구사항 존재

## 💬 실행 프롬프트

```markdown
# [들음] 01.requirement — 요구사항 도출

hearing-들음/req.md의 요구사항을 분석하여 설계 산출물을 생성하라.

⚠️ **왜-어떻게-무엇을 — 황금원 분석 (Golden Circle)**

> "Start with Why." — Simon Sinek  
> Scripture는 Why부터 듣는다. What만 보면 방향을 잃는다.

req.md의 각 기능을 아래 3단계로 분해하라. **이 순서대로 읽어라:**

| 단계 | 질문 | 이번 예시 |
|:--|:--|:--|
| **왜 (Why)** | 사용자가 이것을 왜 원하는가? | 빠른 정보 접근을 위해 |
| **어떻게 (How)** | 사용자가 어떤 행위를 하는가? | 카메라로 QR을 스캔한다 |
| **무엇을 (What)** | 시스템이 무엇을 만드는가? | 스캔된 내용을 화면에 표시 |

⚠️ **What만 보면 죄다.** How(사용자 행위 = 동사)를 먼저 확인하면 방향이 결정된다.

⚠️ **경쟁 해석 소거법 — 가능한 해석을 모두 나열하고 하나씩 제거하라**

> *"Prove all things."* — 1 Thessalonians 5:21

핵심 기능마다 가능한 해석 A, B, C를 전부 나열한 뒤, How(동사) 기준으로 소거하라.
마지막에 생존한 해석만 Assumptions에 기록한다.

```
## 경쟁 해석 소거
| 기능 | 해석 A | 해석 B | 소거 근거 | 채택 |
|:--|:--|:--|:--|:--:|
| {기능} | {해석A} | {해석B} | How(동사) "{동사}" = {근거} | A or B |
```

⚠️ **충돌 요구사항 삼각 검증 — 둘이 충돌할 때 제3의 근거를 찾아라**

> *"In the mouth of two or three witnesses every word may be established."* — Matthew 18:16 (KJV)

req.md에서 두 요구사항이 서로 충돌하거나 모순되는 것처럼 보이면:
두 요구사항만 보고 판단하지 말라. 반드시 **제3의 독립된 근거**(다른 요구사항, 비즈니스 목표, 기술 제약)를 먼저 수집한 뒤 판단하라.
침묵으로 해소하는 것은 할루시네이션의 씨앗이다.

```
## 충돌 요구사항 삼각 검증
| 요구사항 A | 요구사항 B | 제3 독립 근거 | 해소 방향 |
|:--|:--|:--|:--|
| {충돌 A} | {충돌 B} | {제3 근거 — 다른 req 또는 비즈니스 목표} | {해소 결론} |
```
충돌 건수 0건이면 이 표를 생략하라.

⚠️ **기능 고유성 전수 조사 — 각 REQ는 고유한 기능을 가리키는가?**

도출된 REQ 목록을 전수 조사하라. 두 REQ가 실질적으로 동일한 기능을 가리키거나 겹치는 범위가 있으면 병합하거나 경계를 명확히 나눠라.
"비슷한 기능이니 같은 REQ에 넣어도 되겠지"는 할루시네이션의 출발점이다.

```
## 기능 고유성 검증
| REQ-ID | 기능 요약 | 다른 REQ와 기능이 겹치는가? | 처리 |
|:--|:--|:--|:--|
| REQ-001 | {기능 한 줄 요약} | ❌ 없음 / ⚠️ REQ-XXX와 겹침 | 독립 유지 / 병합 / 경계 분리 |
```

⚠️ **어의 중의성 적발 — 같은 단어가 다른 의미로 쓰였는가?**

req.md에서 **한 단어가 문맥에 따라 다른 의미**로 해석될 수 있는 경우를 찾아라.
중의성 발견 시 → Open Questions에 기록하고 가장 문자적 의미로 채택하라.

⚠️ **핵심 동사 오독 검증 — 동사를 명사로 읽지 않았는가?**

req.md의 **핵심 동사**(행위어)를 먼저 식별하고, 그 동사가 명사구로 오독되지 않았는지 검증하라.
동사 = 사용자가 하는 행위 = 시스템이 구현해야 할 기능의 방향이다.

⚠️ **주어-행위 방향 검증 — 누가 무엇을 하는가?**

req.md에서 **주어(누가)**와 **행위 방향**을 명확히 분리하라.
사용자가 하는 행위인지, 시스템이 하는 행위인지 혼동하면 방향이 반대로 된다.

```
올바른 방향 분석:
주어: 사용자 / 행위: QR을 스캔한다 / 시스템: 결과를 표시
잘못된 방향: 시스템이 QR을 생성하여 표시 (주어와 행위 역전)
```

⚠️ **재료 검수 (Smart Validation)**
req.md를 분석하여 아래 필수 정보를 확인하라:
[ ] 1. 프로젝트의 핵심 목적과 비즈니스 목표
[ ] 2. 주요 기능 목록
[ ] 3. 기술 스택 (백엔드, 프론트, DB)
[ ] 4. 배포 환경

누락 항목이 있으면 번호 선택형 메뉴를 제시하라. 개방형 질문 금지.

⚠️ **가정 선언 — 내가 전하는 복음을 먼저 선언하라 (갈 1:8)**

> "내가 전한 복음을 다르게 전하면 저주를 받는다." — 갈라디아서 1:8

req.md를 해석하기 전, AI는 반드시 아래 형식으로 **가정을 먼저 명시**하라.
사용자 확인이 없는 자동 실행 조건에서는 **가장 문자 그대로의 해석**을 채택하고 진행하라.
침묵으로 가정하는 것은 죄다.

```
## 나의 해석 가정 (Assumptions)
| # | req.md 문장 | 나의 해석 | 근거 |
|:--|:--|:--|:--|
| 1 | "{req.md 모호한 문장}" | {How(사용자 행위 동사) 기반으로 결정한 해석} | {Why-How-What 분석 근거} |
| 2 | ... | ... | ... |
→ 위 해석이 틀렸다면 즉시 알려주세요.
```

⚠️ **불명확한 말씀 — 열린 질문 (Open Questions)**

> 선지자에게 묻지 않은 가정은 할루시네이션의 씨앗이다.

req.md에서 **중의적이거나 모호한 표현**을 발견하면 반드시 명세서 안에 Open Questions 섹션을 만들고 기록하라.
자동 실행 조건에서는 질문을 기록하되, **가장 일반적이고 문자적인 해석으로 진행**하라.

```
## Open Questions (열린 질문)
| # | 모호한 표현 | 채택한 해석 | 확인 필요 여부 |
|:--|:--|:--|:--|
| 1 | "{req.md 중의적 표현}" | {AI가 채택한 해석} | 확인 권장 |
