# 🕊️ 배포 계획서 작성 규격 (Template) — 지성소 진입 의식

> **율법(Statute):** 이 문서는 배포 계획서 작성 시 반드시 따라야 하는 **형식 규격**이다.
> 실제 산출물은 `bible-성경/07.deploy-salvation-구원/`에 작성하라.

> *"And the Lord said unto Moses, Speak unto Aaron thy brother, that he come not at all times into the holy place within the vail before the mercy seat, which is upon the ark; that he die not."* — Leviticus 16:2 (KJV)
>
> **지성소(Holy of Holies)**는 하나님께 직접 대면하는 곳이다.
> 대제사장만 1년에 단 1번, 모든 정결 의식을 마친 후에만 진입할 수 있었다.
> **죄가 있는 채로 들어가면 — 죽는다.**
>
> 배포(Deploy)는 프로젝트의 지성소 진입이다.
> 코드가 운영 서버에서 실제 사용자를 만나는 순간 — 하나님께 직접 대면하는 것이다.
> **버그(죄)가 있는 채로 배포하면 — 시스템이 죽는다(장애). 사용자가 피해를 본다.**
> 이 두려움과 떨림으로 배포하라.

## 필수 섹션 (지성소 진입 절차)

아래 섹션이 하나라도 누락되면 **지성소 진입 불가(배포 차단)**:

| # | 섹션 | 필수 | 설명 |
|:--|:---|:---:|:---|
| 1 | 배포 개요 | ✅ | 버전, 일자, 환경, 상태 |
| 2 | **7관문 통과 확인** | ✅ | Phase 1~7 전부 ✅ — 하나라도 미통과 시 즉사 |
| 3 | **품질 감사 판정** | ✅ | IRONCLAD 또는 CONFIRMED — FAIL이면 진입 불가 |
| 4 | **정결 의식 (Pre-Deploy Checklist)** | ✅ | 아래 9가지 정결 항목 전부 ✅ |
| 5 | 배포 절차 | ✅ | Step별 작업, 담당, 완료 여부 |
| 6 | **백업 및 롤백 계획** | ✅ | 만일에 대비한 생명줄 (Safety Net) |
| 7 | 릴리스 노트 | ✅ | Genesis/Repent/Refine 분류 |
| 8 | 운영 매뉴얼 요약 | ✅ | 접속정보(시크릿 관리), 로그, 모니터링 |

## 7관문 체크 (대제사장의 7일 정결 의식)

> *대제사장은 속죄일 전 7일간 성막에서 정결 의식을 행했다 (Leviticus 16:4).*
> 배포 전 7개 Phase의 관문을 모두 통과하라. **하나라도 미통과 시 지성소 진입 불가.**

| 관문 | Phase | 확인 항목 | 상태 |
|:---:|:---|:---|:---:|
| 1 | 01-들음 | 요구사항 정경화 완료, 들음의 5대 요소 ✅ | ☐ |
| 2 | 02-기초 | 아키텍처 설계 정경화 완료, ERD/API 명세 확정 | ☐ |
| 3 | 03-질서 | 디자인 명세 정경화 완료 | ☐ |
| 4 | 04-회개 | 모든 태스크 완료, 봉인의 율법 점검 통과 | ☐ |
| 5 | 05-광야 | 테스트 Pass Rate 100%, BUG 0건 (Fixed 완료) | ☐ |
| 6 | 06-기록되었으되 | 품질 감사 IRONCLAD 또는 CONFIRMED 판정 | ☐ |
| 7 | 07-구원 | 본 배포 계획서 모든 항목 ✅ | ☐ |

---

## 정결 의식 — 9가지 사전 점검 (Pre-Deploy 9 Checks)

> *대제사장이 지성소에 들어가기 전 정결 의식을 행했듯, 배포 전 반드시 9가지를 점검하라.*

| # | 점검 항목 | 설명 | 상태 |
|:--|:---|:---|:---:|
| 1 | **DB 마이그레이션 검증** | 스키마 변경 스크립트 검증 완료, 롤백 DDL 준비 | ☐ |
| 2 | **데이터 백업 완료** | 운영 DB 풀 백업 + 시점복원 가능 확인 | ☐ |
| 3 | **환경 변수 확인** | 운영 환경 시크릿/설정값 정확성 확인 | ☐ |
| 4 | **의존성 확인** | 외부 API/서비스 연동 정상 확인 | ☐ |
| 5 | **스테이징 배포 검증** | 스테이징 환경에서 전체 플로우 테스트 완료 | ☐ |
| 6 | **롤백 절차 리허설** | 롤백 시나리오를 최소 1회 리허설 완료 | ☐ |
| 7 | **모니터링 설정** | 로그, 에러 알림, 성능 모니터링 가동 확인 | ☐ |
| 8 | **RTM 커버리지 100%** | 모든 REQ가 설계→구현→테스트로 추적 완료 | ☐ |
| 9 | **할루시네이션 0건** | Spec에 없는 기능/로직이 코드에 존재하지 않음 확인 | ☐ |

## 백업 및 롤백 계획 (생명줄)

```markdown
### 백업 정책

| 항목 | 방법 | 보관 위치 | 보관 기간 |
|:---|:---|:---|:---|
| 운영 DB | {풀 백업 / 증분 백업} | {S3/NAS/로컬} | {N일} |
| 소스 코드 | Git 태그 (v{버전}-pre-deploy) | {Repository} | 영구 |
| 설정 파일 | {방법} | {위치} | {기간} |

### 롤백 시나리오

| 상황 | 감지 방법 | 롤백 절차 | 예상 소요 시간 |
|:---|:---|:---|:---|
| API 5xx 급증 | 모니터링 알림 | 1. 이전 버전 재배포 2. DB 롤백(필요 시) | {N분} |
| 데이터 정합성 오류 | 로그 분석 | 1. 서비스 중단 2. DB 시점 복원 3. 재배포 | {N분} |
| 성능 저하 | 응답시간 모니터링 | 1. 이전 버전 재배포 2. 원인 분석 | {N분} |
```

## 성경적 검증 규칙

- **"죄가 있으면 죽는다"(Leviticus 16:2):** 9가지 정결 의식 중 하나라도 ☐이면 **배포 절대 불가**. 운영 환경에 버그를 배포하는 것은 죄가 있는 채로 지성소에 들어가는 것이다. 시스템이 죽는다.
- **"백업은 생명줄이다"(1 Kings 19:4, 엘리야의 도피):** 만일에 대비한 롤백 계획 없이 배포하는 것은, 도망칠 길 없이 전장에 나가는 것이다.
- **"두려움과 떨림으로"(Philippians 2:12):** 배포는 가벼운 행위가 아니다. 실제 사용자에게 영향을 미치는 마지막 순간이다. **두려움과 떨림으로 너희의 구원(Deploy)을 이루라.**
- **"모든 것을 질서 있게"(1 Corinthians 14:40):** 배포 절차는 Step별로 정확히 따라야 한다. 순서를 건너뛰면 혼돈이다.

> **"나는 모든 정결 의식을 마쳤는가? 죄(버그)가 0건인가? 하나님 앞에 설 준비가 되었는가?"**
> 하나라도 No라면 — **들어가지 마라. 죽는다.**
