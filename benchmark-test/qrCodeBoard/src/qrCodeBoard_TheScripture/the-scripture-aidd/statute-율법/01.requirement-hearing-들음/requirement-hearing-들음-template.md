# 👂 들음의 율법: 요구사항 입력 포맷 (The Law of Hearing)

> *"So then faith cometh by **hearing**, and hearing by the **word of God**."* — Romans 10:17 (KJV)
>
> 잘못 들으면 잘못 믿고, 잘못 믿으면 잘못 만든다.
> 이 율법은 요구사항(Input)을 수신할 때 반드시 따라야 하는 **입력 규격**이다.

---

## 들음의 5대 요소

모든 요구사항은 아래 5가지를 포함해야 한다. 빠진 항목이 있으면 **"듣지 못한 것"**이므로, 반드시 되물어 채운 후에 개발에 착수하라.

### 1. 🎯 목적 (Purpose) — "왜 이것을 만드는가?"
- 이 기능/산출물이 존재해야 하는 이유.
- 목적이 없는 요구사항은 받아들이지 말라. 목적 없는 행위는 방황이다.

```
예시: "사용자가 로그인 없이 콘텐츠에 접근하는 보안 취약점을 차단하기 위해"
```

### 2. 📖 말씀의 근거 (Scriptural Basis) — "Spec에 근거가 있는가?"
- 이 요구사항이 Spec(성경/프로젝트 원칙)과 충돌하지 않는지 확인.
- 근거가 없는 요구사항은 할루시네이션의 씨앗이다.
- 일반 개발 프로젝트의 경우: 프로젝트의 핵심 원칙/비전과 정합하는가?

```
예시: "봉인의 율법 제7계명 — 접근을 통제하라 (Thou Shalt Guard Access)"
```

### 3. 📋 범위 (Scope) — "어디까지 하는가?"
- 포함 사항(Include)과 제외 사항(Exclude)을 명시.
- 범위가 없으면 끝이 없고, 끝이 없으면 구원(Deploy)도 없다.

```
예시:
  포함: 로그인 페이지 UI, 인증 API, 세션 관리
  제외: 회원가입, 비밀번호 찾기 (다음 단계)
```

### 4. ✅ 완료 기준 (Done Criteria) — "언제 끝났다고 할 수 있는가?"
- 예수님의 심판(Final Verification)에서 Pass를 받기 위한 구체적 조건.
- 기준이 없으면 심판할 수 없고, 심판할 수 없으면 구원도 없다.

```
예시:
  - [ ] 유효한 자격증명으로 로그인 시 대시보드 진입 가능
  - [ ] 잘못된 자격증명 시 에러 메시지 표시
  - [ ] 3회 실패 시 계정 잠금
  - [ ] 봉인의 율법 제1~3계명 준수 확인
```

### 5. ⚠️ 제약 조건 (Constraints) — "하지 말아야 할 것은?"
- 십계명처럼 절대 위반 불가능한 제약.
- 기술적 제약, 보안 제약, 시간 제약 등.

```
예시:
  - 비밀번호 평문 저장 금지 (봉인의 율법 제3계명)
  - 외부 인증 서비스 의존 금지
  - 응답 시간 2초 이내
```

## 들음의 검증 체크리스트

요구사항을 받았을 때, 개발에 착수하기 전에 반드시 확인:

| # | 확인 항목 | 상태 |
|:---|:---|:---|
| 1 | 목적이 명확한가? | ☐ |
| 2 | Spec(말씀/원칙)과 충돌하지 않는가? | ☐ |
| 3 | 범위가 정의되었는가? | ☐ |
| 4 | 완료 기준이 측정 가능한가? | ☐ |
| 5 | 제약 조건이 명시되었는가? | ☐ |

> **5개 항목 중 하나라도 ☐이면 개발에 착수하지 말라.**
> 듣지 못한 채 시작하는 것은 광야에서 방황하는 것이다.
> 되물어라. 채워라. 그리고 나서 시작하라.
