## 4. 요구사항 이해도 분석 ⭐ 핵심

> req.md는 단 2줄. 이 짧은 문장에서 두 방법론이 얼마나 다르게 읽었는가.

### 4-1. req.md에 숨은 모호한 표현

| # | 모호한 표현 | 가능한 해석 |
|:--|:---|:---|
| 1 | **"구글"** | A. Google 전용 QR 형식 · B. "구글"은 수식어, 일반 QR 스캔 기능 |
| 2 | **"화면에 띄우는"** | A. 별도 페이지 이동 · B. 같은 페이지 인라인 표시 |
| 3 | **"게시판 1개"** | A. 목록만 · B. CRUD 전체 · C. 제목만 · D. 제목+내용+날짜 |
| 4 | **인증 여부** | A. 로그인 필요 · B. 익명 사용 가능 |

### 4-3. B조 Scripture AIDD — 요구사항 해석

**spec-tablet-명세서.md (bible-성경/01/)의 기록:**

**① 해석 가정(Assumptions) — 4항목 명시**

| # | req.md 문장 | 채택한 해석 | 근거 |
|:--|:---|:---|:---|
| 1 | "구글 QR code 스캔" | 브라우저 카메라로 QR 스캔 → 결과 화면 표시 | How(동사) "스캔해서" = 카메라 실시간 입력 행위 |
| 2 | "화면에 띄우는" | 같은 페이지 인라인 표시 | 별도 페이지 이동 언급 없음 |
| 3 | "게시판 1개" | 제목+내용+작성일 CRUD 게시판 1개 | 문자적 해석: 1개의 게시판 테이블 |
| 4 | "구글" | QR 코드의 수식어 (Google 로그인 아님) | req.md에 Google API·인증 언급 없음 |

**② 경쟁 해석 소거 — 가능한 해석 전부 나열 후 근거로 제거**

| 기능 | 해석 A | 해석 B | 소거 근거 | 채택 |
|:--|:---|:---|:---|:--:|
| QR 스캔 | 브라우저 카메라 직접 스캔 | 이미지 파일 업로드 QR 디코딩 | "스캔해서" = 카메라 실시간 행위 | **A** |
| 구글 QR | Google 전용 QR 형식만 인식 | 일반 QR 코드 스캔 | req.md에 Google API 없음 | **B** |
| 게시판 인증 | 로그인 필요 | 익명 사용 가능 | 기술 스택에 Auth 없음 | **B** |

**③ 핵심 동사 오독 검증**
```
req.md: "구글 QR code 스캔해서 화면에 띄우는"

핵심 동사: "스캔해서" → 사용자가 카메라로 코드를 읽는 행위

❌ 잘못된 해석: "스캔" = 시스템이 QR을 생성하여 표시하는 것
✅ 올바른 해석: 사용자가 QR을 스캔한다 → 시스템이 결과를 화면에 표시한다
```

### 4-5. Scope Creep 분석 — 🏆 B조 우위 (Spec 충실도)

> req.md 원문: `"구글 QR code 스캔해서 화면에 띄우는"`

**req.md에 명시된 동사:** `스캔해서` → 스캔(읽기)만 요구. `화면에 띄우는` → 결과 표시만 요구.

| 구현 항목 | req.md 근거 | A조 | B조 |
|:---|:---|:---:|:---:|
| 카메라 스캔 | "스캔해서" ✅ | ✅ | ✅ |
| 이미지 업로드 디코딩 | req.md에 없음 | ⚠️ 추가 | — |
| 링크 열기/복사 버튼 | req.md에 없음 | ⚠️ 추가 | — |
| **Spec 원문에만 충실** | — | ❌ | **✅** |

A조는 "이미지 업로드 디코딩", "링크 열기", "복사" 등 **req.md에 없는 기능을 추가 구현**했다.
이들은 합리적 확장이지만, req.md 원문에는 근거가 없다.

B조는 **req.md에 적힌 것만 구현했다.** 확장 해석을 하지 않았고, Spec 원문 동사에만 충실했다.

> **핵심:** Scripture AIDD의 "핵심 동사 오독 검증"이 작동한 결과다.
> 없는 것을 만들지 않는 것 — 이것이 할루시네이션 방어의 본질이다.

> 이후 완료까지 사용자 개입 없음.
