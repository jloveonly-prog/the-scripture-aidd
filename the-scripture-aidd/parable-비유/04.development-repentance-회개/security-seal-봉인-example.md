# 🔒 보안 점검 결과 — 성경 검색 앱 (Example)

> **비유(Parable):** 이 문서는 실제 프로젝트에서 봉인의 율법(보안 점검)이 어떻게 적용되는지 보여주는 **참조 예시**이다.
> 실제 산출물 작성 시 이것을 참고하고, 규격은 `statute-율법/04/security-seal-봉인.md`를 따르라.

## 프로젝트: The Scripture Search (성경 검색 앱)

## 봉인의 율법 10계명 점검 결과

| # | 계명 | 점검 항목 | 적용 내용 | 상태 |
|:--|:---|:---|:---|:---:|
| 1 | 입력을 정결케 하라 | 모든 사용자 입력 검증 | 검색어: 2~100자 제한, XSS 필터링 적용 | ✅ |
| 2 | 비밀을 누설하지 말라 | 민감정보 노출 방지 | API 응답에 password_hash 제외, 에러 메시지에 스택트레이스 미포함 | ✅ |
| 3 | 자물쇠를 견고히 하라 | 비밀번호 암호화 | bcrypt(cost=12) 해싱 적용, 평문 저장 0건 | ✅ |
| 4 | 신분을 확인하라 | 인증 체계 | JWT Bearer Token, 만료 시간 1시간, Refresh Token 7일 | ✅ |
| 5 | 허락된 자만 들이라 | 인가/권한 제어 | 북마크 API에 인증 미들웨어 적용, 타인의 북마크 접근 차단 | ✅ |
| 6 | 주사위를 던지지 말라 | 예측 불가 값 사용 | JWT 시크릿: 256bit 랜덤 생성, 세션 ID: uuid v4 | ✅ |
| 7 | 접근을 통제하라 | 접근 제어 | Rate Limiting: 분당 60회, CORS: 허용 도메인 제한 | ✅ |
| 8 | 흔적을 기록하라 | 로깅/감사 | 로그인 시도(성공/실패) 로깅, 비정상 접근 알림 설정 | ✅ |
| 9 | 약한 고리를 찾으라 | 의존성 보안 | npm audit 0 취약점, 의존성 버전 최신 확인 | ✅ |
| 10 | 성벽을 시험하라 | 침투 테스트 | SQL Injection 테스트 통과, CSRF 토큰 적용 | ✅ |

---

## 보안 아키텍처 요약

```
Client → [HTTPS/TLS 1.3] → Nginx(Reverse Proxy) → [Rate Limiter] → API Server
                                                                        │
                                                      [JWT Middleware] ←┘
                                                             │
                                                    [bcrypt] → PostgreSQL
```

## 발견된 취약점 및 조치

| # | 취약점 | 심각도 | 조치 | 상태 |
|:--|:---|:---:|:---|:---:|
| SEC-001 | 검색 API에 Rate Limiting 미적용 | Medium | 분당 60회 제한 적용 | ✅ Fixed |
| SEC-002 | JWT 시크릿이 env 파일에 하드코딩 | High | 환경변수로 외부화 | ✅ Fixed |

> ✅ 봉인의 율법 10계명 전항 Pass. 보안 봉인 완료.
