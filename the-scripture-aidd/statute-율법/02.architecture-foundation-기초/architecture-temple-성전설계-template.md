# 🏗️ 시스템 설계서 작성 규격 (Template)

> **율법(Statute):** 이 문서는 시스템 설계서 작성 시 반드시 따라야 하는 **형식 규격**이다.

---

## 필수 섹션

| # | 섹션 | 필수 | 설명 |
|:--|:---|:---:|:---|
| 1 | 시스템 개요 | ✅ | 시스템명, 버전, 아키텍트, 상태 |
| 2 | 아키텍처 다이어그램 | ✅ | 전체 구조도 (Mermaid 또는 이미지) |
| 3 | 기술 스택 | ✅ | 계층별 기술 + 선택 근거 |
| 4 | **폴더 구조 판단** | ✅ | 아래 판단 체크리스트 수행 후 결과 기록 |
| 5 | 데이터 아키텍처 (ERD) | ✅ | 테이블 정의 + 관계도 |
| 6 | API 명세 | ✅ | Method, Endpoint, 관련 REQ |
| 7 | 시퀀스 다이어그램 | ✅ | 주요 흐름 최소 1개 |
| 8 | 보안 아키텍처 | ✅ | 봉인의 율법 연동 필수 |

## ID 규칙

| 대상 | 접두어 | 예시 |
|:---|:---|:---|
| 아키텍처 컴포넌트 | ARCH- | ARCH-001 |
| API | API- | API-001 |
| 테이블 | TBL- | TBL-001 |

## ⭐ 폴더 구조 판단 체크리스트 (언어 무관)

> *"분별하지 않으면 혼잡이 온다."* — 구조 결정은 코딩 전에 확정하라.

### 판단 규칙 — 아래 체크리스트를 위에서 아래로 순서대로 적용

**먼저 확인: CRUD가 포함되어 있는가?**

```
CRUD(Create/Read/Update/Delete)가 포함된 기능이 있는가?
  → YES: 아래 "CRUD 강제 분리 규칙" 적용 → 반드시 HTTP 처리와 렌더링 분리
  → NO:  "일반 판단표"로 이동
```

### ⚠️ CRUD 강제 분리 규칙

> **CRUD는 목록·작성·상세·수정 = 각각 독립된 화면이다. 이것은 "단순"이 아니다.**
> CRUD가 포함된 기능은 화면 수에 관계없이 **아래 4개 역할을 별도 파일로 분리한다.**

| 역할 | 책임 | 포함해야 하는 것 | 포함하면 ❌인 것 |
|:---|:---|:---|:---|
| **HTTP 핸들러** | 요청 수신 → 응답 반환 | 파라미터 파싱, 상태코드 결정, 서비스 호출 | HTML 렌더링, DB 쿼리 |
| **화면 템플릿** | UI 렌더링 | HTML 구조, 조건부 표시, 데이터 바인딩 | HTTP 처리, 비즈니스 로직 |
| **서비스/비즈니스** | 비즈니스 규칙 | 검증, 계산, 트랜잭션 | HTTP 처리, HTML 렌더링 |
| **데이터/DB** | 데이터 저장/조회 | SQL, ORM 쿼리, 스키마 정의 | HTTP 처리, HTML 렌더링 |

> AI는 Phase 2에서 확정된 기술 스택의 공식 컨벤션에 따라 각 역할의 파일명/폴더명을 결정하라.
> 어떤 언어든 **한 파일에 2개 이상의 역할이 섞이면 분리하라.**

> ⚠️ **"HTTP 핸들러 안에 HTML을 넣으면 편하다"는 마귀의 속삭임이다.**
> CRUD의 각 화면은 HTTP 핸들러와 독립된 책임이다. 한 파일에 넣으면 **반드시 수백 줄로 비대해진다.**

### 일반 판단표 (CRUD가 아닌 경우에만 적용)

| 판단 기준 | 분리 권장 | 통합 허용 |
|:--|:--|:--|
| 화면 수가 **4개 이상**인가? | ✅ 분리 | |
| 화면 수가 **3개 이하**이고 각 화면이 50줄 미만인가? | | ✅ 통합 허용 |
| 단일 랜딩 페이지(화면 1개)인가? | | ✅ 통합 허용 |

### 판단 결과 기록 형식 (반드시 작성)

```markdown
## 폴더 구조 판단

### 판단 체크리스트
| # | 질문 | 답변 | 결과 |
|:--|:---|:---|:---|
| 1 | CRUD가 포함되어 있는가? | ✅ YES / ❌ NO | 분리 강제 / 일반 판단표 |
| 2 | 화면 수는 몇 개인가? (CRUD 각 화면 별도 카운트) | {N}개 | |
| 3 | 한 파일에 HTTP 처리 + 렌더링이 섞이는가? | ✅ YES → 분리 필수 | |

### 판단 결과
- **채택 구조:** {routes/ + views/ 분리 OR routes에 JSX 통합}
- **판단 근거:** {위 체크리스트 결과 요약}
- **파일 목록:**
  - HTTP 처리: {routes/board.ts 등}
  - 렌더링: {views/board/list.tsx, detail.tsx, form.tsx 등}
```

## 검증 규칙

- 모든 ARCH-ID는 REQ-ID로 역추적 가능해야 함 (RTM 등재)
- 보안 아키텍처는 봉인의 율법 제2,3,7계명 최소 준수
- ERD의 모든 테이블은 목적이 명시되어야 함
- **폴더 구조 판단 체크리스트가 빠짐없이 작성되어야 함**

> **위반 시:** 기초(Foundation) 없는 건물은 무너진다. 정경화 거부.
