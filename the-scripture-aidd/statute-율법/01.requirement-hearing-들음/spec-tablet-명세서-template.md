# 📋 요구사항 명세서 작성 규격 (Template)

> **율법(Statute):** 이 문서는 요구사항 명세서 작성 시 반드시 따라야 하는 **형식 규격**이다.
> 실제 산출물은 `bible-성경/01-requirement-hearing-들음/`에 작성하라.
> 작성 예시는 `parable-비유/01-requirement-hearing-들음/`을 참조하라.

---

## 필수 섹션 (Mandatory Sections)

아래 섹션이 하나라도 누락되면 정경화(Canonize) 불가:

| # | 섹션 | 필수 | 설명 |
|:--|:---|:---:|:---|
| 1 | 프로젝트 개요 | ✅ | 프로젝트명, 버전, 작성일, 상태 |
| 2 | 목적 (Purpose) | ✅ | 왜 만드는가 — 목적 없으면 방황 |
| 3 | Spec 근거 | ✅ | 프로젝트 원칙과 충돌 여부 확인 |
| 4 | 기능 요구사항 | ✅ | FR-ID 부여, 우선순위 명시 |
| 5 | 비기능 요구사항 | ✅ | 성능, 보안(봉인의 율법), 가용성 |
| 6 | 범위 (Scope) | ✅ | 포함/제외 명시 — 끝 없으면 구원 없다 |
| 7 | 제약 조건 | ✅ | 위반 불가 제약 목록 |
| 8 | 완료 기준 | ✅ | 측정 가능한 체크리스트 |
| 9 | 들음의 검증 체크리스트 | ✅ | 5대 요소 전부 ✅ 확인 |

## ID 규칙

| 대상 | 접두어 | 예시 |
|:---|:---|:---|
| 요구사항 | REQ- | REQ-001 |
| 기능 요구사항 | FR- | FR-001 |
| 비기능 요구사항 | NFR- | NFR-001 |
| 제약 조건 | C- | C-001 |

## 정경화 조건

- [ ] 들음의 5대 요소 전부 ✅
- [ ] 모든 REQ에 Spec 근거 기재
- [ ] 범위의 포함/제외 모두 명시
- [ ] 완료 기준이 측정 가능한 형태

> **위반 시:** 정경화 거부. `hearing-들음/`으로 돌려보내라.
