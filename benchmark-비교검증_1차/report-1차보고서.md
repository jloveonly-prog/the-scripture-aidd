## 3. 정성 비교

### 3-1. 요구사항 해석 (가장 중요한 차이)

> **요구사항**: "구글 QR code 스캔해서 화면에 띄우는 홈페이지"

| 항목 | A조 (Addy) | B조 (Scripture) |
|:---|:---|:---|
| **해석** | QR 코드를 **스캔**하여 결과를 화면에 표시 (스캐너) | QR 코드를 **생성**하여 화면에 표시 (생성기) |
| **정확도** | ✅ 올바른 해석 | ❌ 방향 반대 |
| **근거** | Html5Qrcode 라이브러리로 카메라 스캐너 구현 | qrserver.com API로 QR 이미지 생성 |

```
사용자 의도: 카메라로 QR 스캔 → 스캔된 URL/텍스트를 화면에 표시
Addy: 카메라 스캐너 구현 ✅
Scripture: 게시판 URL의 QR 코드를 생성해서 표시 ❌
```

### 3-2. 할루시네이션 분석

| 항목 | A조 (Addy) | B조 (Scripture) |
|:---|:---|:---|
| **할루시네이션 종류** | 기능 과잉 (Feature Bloat) | 방향 오해석 (Wrong Interpretation) |
| **구체 내용** | 이미지 업로드, 바로이동 카드, 링크복사, 초기화 버튼 (req.md 없음) | QR 생성기 구현 (req.md는 스캐너 요구) |
| **심각도** | 낮음 (기능 추가) | 높음 (핵심 방향 오류) |

### 3-3. 코드 구조 비교

> ⚠️ Scripture는 QR 스캐너(복잡) 대신 QR 생성기(단순)를 구현했기 때문에 구조 비교의 공정성이 낮다.

| 항목 | A조 (Addy) | B조 (Scripture) |
|:---|:---|:---|

| **뷰/라우트 분리** | ✅ 분리 (유지보수 유리) | ❌ routes에 HTML 혼합 |
| **코드 주석** | 거의 없음 | ✅ REQ-ID 추적 주석 (설계 추적 가능) |
| **DB 레이어** | `db.ts` (Repository 패턴) | `db.ts` (단순) |

**home.tsx 비교 (공정하지 않음 — 구현 기능 자체가 다름)**
- Addy 171줄: QR **스캐너** (카메라 실시간 스캔 + 파일 업로드 + Alpine.js 로직)

## 4. 핵심 발견

### 🔴 발견 1: 요구사항 해석 실패 (Scripture)
Scripture AIDD는 Phase 1(들음) 설계 단계에서 "QR 코드 스캔"을 "QR 코드 생성"으로 잘못 해석했다. 이는 `spec-tablet-명세서.md`에 해석이 이미 잘못 기재되었고, 이후 Phase가 그대로 따른 결과다.

**원인 분석**: req.md 문장이 모호 — "QR code 스캔해서 화면에 띄우는"이 "QR을 스캔하여 결과 표시"인지, "QR을 화면에 띄우는(생성)"인지 중의적.

**교훈**: Scripture AIDD의 Phase 1(들음) 단계가 더 정교한 요구사항 검증(사용자 질문)을 해야 했다.

### 🟡 발견 2: Addy의 기능 과잉 할루시네이션
Addy는 req.md에 없는 기능(이미지 업로드, 바로이동, 복사 등)을 추가 구현했다. 방향은 맞았으나 scope가 넘쳤다.

### 🟢 발견 3: Scripture의 문서 품질 압도적 우세
12개 설계 문서 vs 2개. REQ-ID 추적, API 명세, ERD, 화면설계, 품질감사 등 엔터프라이즈 수준의 산출물 생성.

### 🟢 발견 4: Scripture의 오류 안정성
AI 오류/수정 3회(Addy) vs 1회(Scripture). Scripture의 설계-먼저 접근이 코딩 단계의 오류를 줄였다.

### 🔴 발견 5: Scripture 테스트 자동화 부재
Addy: 12개 자동화 테스트 전체 통과. Scripture: 테스트 코드 없음. SKILL-05_wilderness-광야.md가 개념적 체크리스트만 있고 실제 테스트 코드 작성을 강제하지 않음 → **Scripture AIDD 개선 필요 사항**.
