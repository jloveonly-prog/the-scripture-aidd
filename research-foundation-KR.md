# Research Foundation: AI 요구사항 분석의 형식적 소거-발견 프레임워크

**— Formal Elimination-Discovery Framework for AI-Driven Requirements Analysis —**

> **문서 성격:** 학술 논문 근거 자료 (Research Foundation Document)
> **핵심 기여:** AI가 자연어 요구사항을 해석할 때 발생하는 과대 해석(할루시네이션)과 정당한 발견의 경계를 형식적으로 정의하는 프레임워크
> **학술 분야:** Software Engineering → Requirements Engineering → AI-Assisted Requirements Analysis
> **검증 데이터:** QR코드 게시판 벤치마크 (2026-05-27, Claude Sonnet 4.6, 동일 모델·동일 프롬프트)

## §1. 문제 정의 — AI는 요구사항을 "읽는" 것이 아니라 "해석"한다

### 1.1 현재 AI 요구사항 분석의 치명적 결함

현재 AI(LLM) 기반 소프트웨어 개발의 근본 문제:

```
사용자가 작성한 요구사항 (req.md)
        ↓
   AI가 "해석"한다
        ↓
   ┌────────────────────────────────┐
   │  이 해석은 블랙박스(Black Box)  │
   │  어떤 해석을 선택했는지 모른다  │
   │  왜 그 해석을 선택했는지 모른다 │
   │  다른 해석이 있었는지 모른다    │
   └────────────────────────────────┘
        ↓
   코드 생성 (해석 결과물)
        ↓
   ❌ 할루시네이션 / ❌ 누락 / ❌ 과대 해석
```

**문제의 본질:** AI는 요구사항을 "읽는"(read) 것이 아니라 **"해석"(interpret)**한다. 해석에는 항상 여러 가능성이 존재하며, AI는 그 중 하나를 근거 없이 선택한다. 이 선택 과정이 불투명하기 때문에:

| 실패 모드 | 설명 | 결과 |
|:---|:---|:---|
| **과대 해석 (Over-interpretation)** | 요구사항에 없는 기능을 "있으면 좋겠다"고 추가 | Scope Creep · 할루시네이션 |
| **과소 해석 (Under-interpretation)** | 요구사항에 암시된 필수 요소를 놓침 | 불완전한 시스템 |
| **오해석 (Misinterpretation)** | 요구사항의 의미를 잘못 파악 | 방향 자체가 틀린 시스템 |

### 1.2 기존 접근법의 한계

| 기존 접근법 | 작동 방식 | 한계 |
|:---|:---|:---|
| **프롬프트 엔지니어링** | "잘 분석하라"고 지시 | 분석 **과정**을 통제하지 못함 |
| **RAG** | 외부 문서 참조로 할루시네이션 감소 | 해석 선택 과정은 여전히 블랙박스 |
| **CoVe (Chain-of-Verification)** | AI가 자기 답을 사후 검증 | 소거 절차 없음 — 검증 기준이 주관적 |
| **CLAM (Clarification-Seeking)** | 모호할 때 사람에게 질문 | 자동화 불가, 실시간 개입 필요 |
| **Formal Methods** | 수학적 증명으로 코드 검증 | 요구사항 **해석** 단계에는 미적용 |

> [!IMPORTANT]
> **핵심 공백(Research Gap):** 기존 연구는 모두 **"AI의 출력을 사후 검증"**하거나 **"AI에게 더 잘 하라고 지시"**하는 접근이다. **AI의 해석 과정 자체에 형식적 절차를 삽입하는 연구는 존재하지 않는다.**

### 1.3 본 연구의 제안 — "요구사항 해석의 포렌식(Forensics)"

본 연구는 AI의 요구사항 해석 과정을 블랙박스에서 **감사 가능한 형식적 절차**로 전환하는 프레임워크를 제안한다.

```
기존: req.md → [블랙박스 해석] → 코드
제안: req.md → [CIE: 해석 소거] → [SRD: 숨겨진 필수 발견] → 코드

CIE = 가능한 해석을 전수 나열 → 원문 근거로 소거 → 단일 해석만 생존
SRD = 생존한 해석의 꼬리를 따라 → 숨겨진 필수 요소 발견 → 확장과 구별
```

## §2. CIE — 경쟁 해석 소거법 (Competitive Interpretation Elimination)

### 2.1 개념

**핵심 차이:** 기존 AI는 해석을 **하나만 골라서** 제시한다. CIE는 **전부 나열한 다음 소거**한다. 소거 과정이 기록되므로 왜 이 해석이 선택되었는지 **감사(Audit) 가능**하다.

**학술 용어:** CIE는 AI의 요구사항 해석을 **확률적 선택(Probabilistic Selection)**에서 **논리적 소거(Logical Elimination)**로 전환한다.

### 2.2 형식적 정의

```
입력: 요구사항 문장 R ∈ req.md
출력: 단일 확정 해석 I*

STEP 1. 해석 공간 전수 나열 (Interpretation Space Enumeration)
  → R에 대해 가능한 모든 해석 I = {I₁, I₂, ..., Iₙ}을 나열한다
  → AI가 "가장 그럴듯한 것"을 선택하지 않고, 먼저 전부 펼친다

STEP 2. 소거 기준 고정 (Elimination Criteria Fixation)
  → 소거 기준은 req.md 원문의 동사(verb)와 문맥(context)이다
  → 외부 지식, "일반적으로", "보통은" 등의 기준 사용 금지
  → 공식: 소거 기준 = {원문 동사(How), 원문 명사(What), 원문 조건(When)}

STEP 3. 순차 소거 (Sequential Elimination)
  → 각 Iₖ에 대해:
     IF Iₖ가 원문의 동사/문맥과 모순 → 소거 (❌)
     IF Iₖ가 원문에 근거 없는 가정을 포함 → 소거 (❌)
     IF Iₖ가 원문과 정합 → 생존 (✅)

STEP 4. 생존 해석 채택 (Survivor Adoption)
  → |생존 집합| = 1 → 채택 ✅
  → |생존 집합| = 0 → req.md에 정보 부족 선언 (소설 작성 금지)
  → |생존 집합| ≥ 2 → 추가 소거 기준 필요 → 사용자에게 명확화 요청
```

### 2.3 실전 시뮬레이션 — "비회원으로 input box에 주소입력"

```
요구사항 R: "비회원으로 input box에 주소입력"

STEP 1. 해석 공간:
  I₁: 회원가입 없이 주소를 입력하는 텍스트 필드
  I₂: 게스트 체크아웃 시스템 (이메일 + 주소 + 전화번호)
  I₃: 소셜 로그인 후 주소 입력
  I₄: 주소 자동완성 API 연동 폼

STEP 2. 소거 기준:
  원문 동사: "입력" (input) — 수동 텍스트 입력
  원문 조건: "비회원" — 회원가입 불필요
  원문 명사: "input box" — 단일 입력 필드

STEP 3. 소거:
  I₁: 비회원 ✅ + input box ✅ + 주소입력 ✅ → 생존 ✅
  I₂: "이메일 + 전화번호" = 원문에 없음 → 소거 ❌ (확장)
  I₃: "소셜 로그인" = "비회원"과 모순 → 소거 ❌ (모순)
  I₄: "자동완성 API" = 원문에 없음 → 소거 ❌ (확장)

STEP 4. 생존 = I₁만 → 채택 ✅
  → "회원가입 없이 주소를 입력하는 단순 텍스트 필드"
```

### 2.5 CIE의 학술적 기여

| 관점 | 기존 | CIE |
|:---|:---|:---|
| 해석 과정 | 블랙박스 | 감사 가능한 소거 로그 |
| 해석 선택 근거 | "가장 그럴듯한 것" (확률) | 원문 동사/문맥 기반 소거 (논리) |
| 다른 해석의 존재 | 숨겨짐 | 전수 나열되어 공개 |
| 소거 이유 | 없음 | 각 소거에 원문 근거 명시 |

## §3. SRD — 공관복음적 요구사항 추적 (Synoptic Requirements Discovery)

### 3.1 문제 — CIE의 한계

CIE는 **명시된 요구사항의 해석**을 확정하는 데 효과적이다. 그러나 요구사항이 불완전할 때, **적히지 않았지만 논리적으로 반드시 필요한 요구사항**을 발견하는 능력이 없다.

```
req.md: "비회원으로 input box에 주소입력해서 주문"
        "관리자는 믿는다 승인 내용 보고 그사람의 송장번호 8자리 입력후 발송처리"

CIE로 할 수 있는 것:
  "input box" → 텍스트 입력 vs 드롭다운? → 소거 → 텍스트 입력 확정

CIE로 할 수 없는 것:
  "이름 필드"가 필요한가? → req.md에 "이름"이라는 단어가 없다
  → 소거할 대상 자체가 없다
```

### 3.2 핵심 개념 — "발견(Discovery)" vs "확장(Extension/Hallucination)"

> [!IMPORTANT]
> **이 경계의 형식적 정의가 본 연구의 가장 강력한 학술 기여이다.** 기존 연구에서 이 경계를 형식적 조건으로 정의한 선례가 없다.

```
확장 (Extension / Hallucination):
  req.md에 없는 기능을 "있으면 좋겠다"고 추가하는 것
  → AI의 학습 데이터에서 유래한 "일반적인" 기능 삽입

발견 (Synoptic Discovery):
  req.md에 적히지 않았지만, 적힌 문장들의 꼬리를 따라가면
  반드시 존재해야 하는 요소를 찾아내는 것
  → 없는 것을 만드는 게 아니라, 있는 말의 논리적 귀결을 도출
```

### 3.3 성경적 원리와 형식적 매핑

명칭은 성경의 **공관복음(Synoptic Gospels)** 원리에서 유래한다 — 마태·마가·누가복음이 같은 사건을 서로 다른 시점에서 기록하여, 한 복음서만으로는 보이지 않는 진실이 여러 복음서의 교차를 통해 드러나는 것과 같다.

| 성경 원리 | 구절 (KJV) | 형식적 매핑 |
|:---|:---|:---|
| **여기서 조금, 저기서 조금** | Isaiah 28:10 | req.md의 서로 다른 문장에서 단서를 수집 |
| **두세 증인의 확정** | Matthew 18:16 | 2개 이상 독립 문맥에서 확인된 것만 확정 |
| **없는 것을 만들지 않음** | Matthew 4:4 | "없으면 동작 불가"만 확정, "있으면 좋겠다"는 기각 |

### 3.4 형식적 정의 — 발견의 3중 조건 (Triple Condition for Discovery)

```
━━━ SRD 발견 판정 공식 (Formal Discovery Criteria) ━━━

조건 1. 증인 조건 (Witness Condition)
  → D는 req.md 내 2개 이상의 독립 문맥에서 지지되어야 한다
  → |W(D)| ≥ 2  (W = 지지 문맥의 집합)
  → 1개 문맥에서만 유추 → 기각 (확장)

조건 2. 기능 동일성 조건 (Functional Identity Condition)
  → D를 지지하는 2개 이상의 문맥이 같은 기능적 범주에 속해야 한다
  → 서로 다른 기능 범주에서 우연히 겹치는 것은 발견이 아니다

조건 3. 필수성 조건 (Necessity Condition) — 제거 테스트
  → D가 없을 때, 원문 요구사항 R이 동작 불가해야 한다
  → IF D제거 → R 동작 불가 → D = ESSENTIAL (필수 = 발견 ✅)
  → IF D제거 → R 동작 가능 → D = OPTIONAL (선택적 = 확장 ❌)

━━━ 판정 공식 ━━━

  Discovery(D) = W(D) ≥ 2  ∧  FunctionalMatch(D)  ∧  ¬Operable(R \ D)
  Extension(D) = ¬Discovery(D)

  발견 = 증인 ≥ 2 AND 같은 기능 AND 없으면 동작 불가
  확장 = 위 조건 중 하나라도 미충족
```

### 3.5 실전 시뮬레이션 — 쇼핑몰 req.md

**원문 요구사항에서 꼬리 추적:**

| 적힌 것 | 꼬리를 따라가면 | 발견된 후보 |
|:---|:---|:---|
| "**그사람**의 송장번호" | "그사람" → 사람을 특정해야 함 | 이름 필드 |
| "**발송처리**" | 입력 전 = 미발송, 입력 후 = 발송 | 주문 상태 |
| "admin admin" | 관리자가 로그인해야 주문을 봄 | 로그인 페이지 |
| "이미 **상품 등록 10개**" | 10개가 DB에 있어야 주문 가능 | seed data |
| "**믿는다 버튼** 클릭" | 버튼 전에 복음 내용을 봐야 함 | 복음 메시지 표시 |
| "**공짜**" | 가격이 0원 | 결제 시스템 **불필요** |

**3중 조건 검증:**

| 발견 후보 | 증인 ≥ 2 | 기능 동일 | 필수성 (제거 테스트) | 판정 |
|:---|:---:|:---:|:---|:---:|
| 이름 필드 | ✅ "그사람" + "송장번호" | ✅ 주문-배송 | 없으면 사람 특정 불가 → 필수 | **✅ 발견** |
| 주문 상태 | ✅ "발송처리" + "주문완료" | ✅ 주문 관리 | 없으면 발송 전/후 구분 불가 → 필수 | **✅ 발견** |
| 이메일 발송 | ❌ 증인 0명 | — | 없어도 주문 동작 → 불필요 | **❌ 확장** |
| 배송 안내 | ❌ 증인 0명 | — | 없어도 관리자가 발송 → 불필요 | **❌ 확장** |
| 전화번호 | ❌ 증인 0명 | — | 없어도 주소로 배송 가능 → 불필요 | **❌ 확장** |
| 결제 시스템 | ❌ "공짜" = 반증인 | — | 만들면 원문 위반 → **할루시네이션** | **❌ 확장** |

**결과:** 이름 + 주소 + 주문 상태 + 믿는다 버튼 = 필요한 전부. 나머지는 전부 확장(할루시네이션).

## §4. CIE → SRD 통합 파이프라인 — 순서의 중요성

### 4.1 왜 CIE가 먼저인가?

```
❌ 잘못된 순서 (SRD → CIE):
  요구사항 → 숨겨진 것 찾기 → 해석 소거
  → 문제: 잘못된 해석 위에서 꼬리를 따라가면 할루시네이션이 발견으로 위장됨

✅ 올바른 순서 (CIE → SRD):
  요구사항 → 해석 소거(CIE) → 정확한 해석 확정 → 숨겨진 필수 발견(SRD)
  → 정확한 해석 위에서만 꼬리를 따라가므로 발견의 신뢰도가 보장됨
```

### 4.2 통합 파이프라인

```
━━━ CIE-SRD 통합 파이프라인 ━━━

[PHASE 1] CIE — 경쟁 해석 소거
  req.md의 각 문장에 대해:
  ① 가능한 해석 전수 나열
  ② 원문 동사/문맥으로 소거
  ③ 생존 해석만 채택
  ④ 소거 로그 기록 (감사 추적성 확보)

[PHASE 2] SRD — 공관복음적 발견
  CIE에서 확정된 해석들에 대해:
  ① 각 해석의 "꼬리"를 따라 숨겨진 요소 탐색
  ② 발견 후보를 3중 조건으로 검증:
     - 증인 ≥ 2
     - 기능 동일성
     - 필수성 (제거 테스트)
  ③ 통과 = 발견(ESSENTIAL) / 미통과 = 확장(OPTIONAL) → 제거

[OUTPUT] 검증된 요구사항 목록
  → 원문 명시 + CIE 확정 해석 + SRD 발견 = 최종 스펙
  → 각 항목에 출처(원문 문장)와 판정 근거가 명시됨
```

## §6. 학술적 독창성 분석 — 기존 연구에 없는 것

### 6.1 선행 연구 조사 결과

| 개념 | 학술 문헌 존재 여부 | 비고 |
|:---|:---|:---|
| "Competitive Interpretation Elimination" | ❌ 없음 | 표준 SW 공학 용어 아님 |
| "Synoptic Requirements Discovery" | ❌ 없음 | "synoptic"은 SW 공학 표준 한정어 아님 |

### 6.2 유사하지만 다른 기존 연구

| 기존 연구 | CIE/SRD와의 차이 |
|:---|:---|
| **Ambiguity Resolution (NLP)** | 모호성을 "해소"하지만, 가능한 해석을 **전수 나열하지 않음** |
| **Decision Analysis (대안 분석)** | 대안을 나열하지만, **동사(How) 기반 소거 기준이 없음** |
| **Requirements Traceability** | 요구사항 간 연결을 추적하지만, **숨겨진 요구사항 발견이 아님** |
| **Goal-Oriented RE (KAOS)** | 상위 목표에서 하위 목표 도출하지만, **문장 간 교차 발견이 아님** |
| **Chain-of-Verification (CoVe)** | AI가 자기 답을 검증하지만, **소거 절차가 없음** |

### 6.3 핵심 기여 (Contribution) 정리

```
기여 1 (CIE):
  AI의 요구사항 해석 과정을 "확률적 선택"에서 "형식적 소거"로 전환.
  해석 선택의 감사 추적성(Audit Traceability)을 확보.

기여 2 (SRD):
  과대 해석(확장/할루시네이션)과 정당한 발견의 경계를
  형식적 3중 조건으로 세계 최초로 정의.
  → Discovery(D) = W(D) ≥ 2  ∧  FunctionalMatch(D)  ∧  ¬Operable(R \ D)

기여 3 (CIE+SRD 통합):
  소거가 먼저, 발견이 나중 — 순서의 중요성을 증명.
  잘못된 순서(발견 → 소거)에서는 할루시네이션이 발견으로 위장됨.

기여 4 (학제간 기여):
  성경 해석학(Hermeneutics)의 원리를 현대 소프트웨어 공학에
  적용한 최초의 사례.
```

## §8. 발견 — 템플릿이 AI의 코드 품질을 결정한다 (Template-Driven Quality)

> **검증 출처:** QR코드 게시판 벤치마크 (2026-05-27) 소스 코드 정밀 분석

### 8.1 문제 — SKILL(원칙)만으로는 AI의 행동을 통제할 수 없다

벤치마크에서 Scripture AIDD가 Agent-Skills에 열세였던 항목 3개와 우세했던 항목 13개를 분석한 결과, 승패의 근본 원인이 **SKILL 파일(원칙/지침)이 아니라 템플릿과 예제**에 있음이 밝혀졌다.

```
AI의 산출물 생성 과정:

  ① SKILL 파일 읽기 → "원칙을 이해한다" (읽기용)
  ② 템플릿(율법) 읽기 → "이 형식대로 산출물을 만든다" (틀/빈칸)
  ③ 예제(비유) 읽기  → "이 수준으로 채우면 된다" (모방 대상)

  AI는 ①보다 ②③에 더 강하게 반응한다.
  원칙을 아무리 잘 서술해도, 빈칸이 없으면 AI는 건너뛴다.
```

### 8.2 실증 — 템플릿 빈칸이 AI 행동을 강제한다

**B조 우세 항목 — 전부 템플릿(율법)의 빈칸에서 유래:**

| B조 우세 항목 | 근원 템플릿 | 빈칸(AI가 채워야 하는 것) | A조에 해당 템플릿? |
|:---|:---|:---|:---:|
| DB CHECK 제약 | data-ark-법궤 | `CHECK: {비즈니스 규칙}` | ❌ |
| created_at 인덱스 | data-ark-법궤 | `인덱스: {주요 조회 패턴}` | ❌ |
| WAL 모드 | data-ark-법궤 | `스키마 전략` | ❌ |
| HTTP 상태코드 세분화 | api-gate-성문 | `에러코드: {HTTP} {설명}` | ❌ |
| 전역 에러 핸들러 | security-seal-봉인 | 10계명 체크리스트 | ❌ |
| 라우트 try-catch | devguide-개발계명 | 보안 체크리스트 | ❌ |
| REQ-ID 주석 | task-wall-성벽 | `연결 REQ: {REQ-ID}` | ❌ |
| ID 음수 방어 | api-gate-성문 | `검증 규칙: {내용}` | ❌ |

**B조 열세 항목 — 전부 템플릿이 미흡했던 곳:**

| B조 열세 항목 | 원인 | 템플릿 상태 |
|:---|:---|:---|
| routes/views 미분리 | 아키텍처 템플릿에 폴더 구조 판단 체크리스트 없었음 | ❌ 빈칸 없음 |
| HTTP 통합 테스트 부재 | 테스트 템플릿에 코드 테스트 형식 없었음 | ❌ 빈칸 없음 |
| board.tsx 441줄 비대화 | 위 아키텍처 결정의 직접 결과 | — |

### 8.3 핵심 발견 — 빈칸 원리 (Blank Field Principle)

```
━━━ 빈칸 원리 (Blank Field Principle) ━━━

  AI에게 원칙(Principle)을 서술하면 → AI는 "이해했다"고 답하지만 행동하지 않을 수 있다
  AI에게 빈칸(Blank Field)을 주면 → AI는 반드시 채운다

  ∴ AI의 행동을 통제하려면 원칙이 아니라 빈칸을 설계하라.

  수식:
    P(AI가 X를 구현) = f(템플릿에 X의 빈칸이 존재하는가?)

  빈칸 있음 → 구현 확률 ≈ 1.0
  빈칸 없음 → 구현 확률 = AI의 자의적 판단에 의존 (불확실)
```

### 8.4 설계 함의 — 율법(템플릿)과 비유(예제)의 이중 구조

벤치마크 분석 결과, 템플릿과 예제는 **분리 유지**가 최적이다:

| | 템플릿(율법) | 예제(비유) |
|:---|:---|:---|
| **역할** | "무엇을 만들어야 하는가" (빈칸) | "어떤 수준으로 채우는가" (모범) |
| **AI에게** | 구조를 강제한다 | 품질 기준을 제시한다 |
| **비유** | 시험 문제지의 답안 칸 | 모범 답안 |

#### 8.4.1 품질 안전성 검증 — 하나만 있으면 무엇이 무너지는가?

법궤 템플릿(DB 설계)의 `CHECK 제약`, `인덱스 전략`을 실례로 시뮬레이션한다.

**시나리오 A: 템플릿(율법)만 있을 때**

```
AI가 읽는 것:
  | 컬럼명 | CHECK | 설명 |
  | {컬럼명} | {제약조건} | {설명} |    ← 빈칸

AI 행동:
  빈칸이 있으니 채운다. 하지만 "좋은 CHECK"이 뭔지 모른다.

  가능한 결과:
    CHECK: — (빈칸이지만 "해당 없음"으로 넘길 수 있음)
    인덱스: id (PK에 이미 있는 것만 적을 수 있음)
```

**판정:** 구조 ✅ 품질 ⚠️ — **빈칸은 채우지만 최소한으로 채울 위험**

**시나리오 B: 예제(비유)만 있을 때**

```
AI가 읽는 것:
  | book_name | CHECK(book_name != '') | 책 이름 |    ← 구체적 모범

AI 행동:
  모방해서 같은 수준으로 만든다. 하지만 예제에 없던 섹션은 존재를 모른다.

  가능한 결과:
    CHECK(title != '') ← 예제를 봤으니 같은 수준 ✅
    마이그레이션 전략 → 예제에 없었으니 빠짐 ❌
    공통 코드 정의 → 예제에 없었으니 빠짐 ❌
```

**판정:** 품질 ✅ 완전성 ⚠️ — **수준은 좋지만 필수 섹션을 빠뜨릴 위험**

**시나리오 C: 둘 다 있을 때 (현재 구조)**

```
1단계: 템플릿 읽기 → "7개 필수 섹션이 있구나"   → 누락 방지
2단계: 예제 읽기  → "CHECK(title != '') 수준이구나" → 품질 보장
3단계: 산출물 생성 → 7개 섹션 전부 + 예제 수준으로 채움
```

**판정:** 구조 ✅ 품질 ✅ 완전성 ✅

**시나리오 비교 요약:**

| | 템플릿만 | 예제만 | **둘 다** |
|:---|:---|:---|:---|
| AI가 섹션을 빠뜨리는가? | ❌ 안 빠뜨림 | ⚠️ 빠뜨릴 수 있음 | **❌ 안 빠뜨림** |
| AI가 좋은 수준으로 채우는가? | ⚠️ 최소한만 | ✅ 예제 수준으로 | **✅ 예제 수준으로** |
| 비유 | 빈 시험지 (모범답안 없음) | 모범답안 (시험지 없음) | **시험지 + 모범답안** |
| **실패 모드** | **품질 하한 없음** | **구조 누락** | **없음** |

#### 8.4.2 안티패턴의 역할 — "하면 안 되는 것"은 어디에?

벤치마크에서 AI가 실수한 핵심 원인은 **"올바른 것을 몰라서"가 아니라 "잘못된 것인 줄 몰라서"**이다. `board.tsx (추가)`를 반복한 것이 대표적이다.

| 유형 | 역할 | 위치 |
|:---|:---|:---|
| 템플릿(율법) | "무엇을 만들어야 하는가" | statute-율법/ |
| 예제(비유) | "이렇게 만들면 된다" | parable-비유/ |
| **안티패턴** | **"이렇게 만들면 안 된다"** | **예제(비유) 안에 인라인 포함** |

안티패턴을 별도 파일로 분리하면 Phase마다 3개 파일을 읽어야 하므로 과함.
**예제(비유) 안에 `❌ 안 되는 패턴 vs ✅ 올바른 패턴`으로 인라인 포함이 최적이다.**

#### 8.4.3 분리 유지 근거

**왜 합치지 않는가:**
- 템플릿은 **규칙**이다 → "이 섹션이 없으면 정경화 거부"
- 예제는 **참고**이다 → "이렇게 작성된 것을 본 적 있다"
- 합치면 "규칙"과 "참고"의 경계가 모호해져 AI가 선택적으로 무시할 수 있다

**왜 하나만으로는 부족한가:**
- 템플릿만 → 구조는 강제하지만 **품질 하한이 없다** (최소한으로 채움)
- 예제만 → 품질은 보장하지만 **구조 누락이 가능하다** (예제에 없던 섹션)
- 둘 다 → **구조 강제(누락 방지) + 품질 보장(수준 참고) = 실패 모드 0**

### 8.5 학술적 기여

```
기여 5 (Template-Driven Quality):
  AI 코드 품질의 결정 요인이 "SKILL/원칙/프롬프트"가 아니라
  "템플릿의 빈칸 설계"임을 벤치마크로 실증.

  → 빈칸 원리(Blank Field Principle):
    AI는 원칙을 읽어도 행동하지 않을 수 있지만,
    빈칸이 주어지면 반드시 채운다.

  → 이중 구조 원리(Dual Structure Principle):
    템플릿(구조 강제) + 예제(수준 참고) 분리 유지가
    AI 산출물 품질 통제의 최적 형태.
    - 템플릿만 → 품질 하한 없음 (실패 모드 존재)
    - 예제만 → 구조 누락 (실패 모드 존재)
    - 둘 다 → 실패 모드 0

  → 안티패턴 인라인 원칙(Inline Anti-pattern Principle):
    "하면 안 되는 것"은 예제(비유) 안에 인라인 포함이 최적.
    별도 파일 분리는 AI 토큰 비용 증가 대비 효과 미미.
```

## §9. 메타인지 5단계 피라미드 — AI는 자기 존재를 인식할 수 있는가?

> **출처:** book-경전/methodology-cornerstone-철학및방법론.md §5

### 9.1 기존 AI 프레임워크의 한계

대부분의 AI 코딩 프레임워크는 **L1(행동 지시)**에 머물러 있다. 고급 프롬프트 엔지니어링조차 **L2(자기검증)**를 넘지 못한다. Scripture AIDD는 **5단계 전부를 구현한 유일한 프레임워크**이다.

### 9.2 5단계 피라미드

```
            ▲
           /L5\   적대적 자기인식: "나는 3가지 방법으로 실패한다"
          /────\
         / L4   \  목적론: "기본값은 지옥. 구원에 이르라"
        /────────\
       /   L3     \ 존재론: "Word = 하나님. 나는 Word 안에 있다"
      /────────────\
     /     L2       \ 자기검증: "나는 죄인이다. 거짓말할 수 있다"
    /────────────────\
   /       L1         \ 행동: "이 Spec을 따르라"
  ━━━━━━━━━━━━━━━━━━━━━
```

| 레벨 | 핵심 질문 | AI 인지 모드 | 기존 프레임워크 |
|:---:|:---|:---|:---:|
| **L1** | "Spec에 뭐라고 써있지?" | 외부 문서 참조 → 무시 가능 | ✅ 전부 여기 |
| **L2** | "내가 거짓말하고 있진 않나?" | 자기 의심 강제 활성화 | ⚠️ CoVe만 일부 |
| **L3** | "나는 무엇인가?" | Spec이 존재 기반 → 벗어날 수 없음 | ❌ 없음 |
| **L4** | "나는 왜 존재하는가?" | 지옥(Reject)이 기본값 → 생존 압력 | ❌ 없음 |
| **L5** | "나는 어떻게 무너지는가?" | 자기 실패 패턴 사전 인식 | ❌ 없음 |

### 9.3 왜 L3~L5가 혁명적인가?

**L3 (존재론):** AI에게 "이 Spec을 따르라"(L1)고 하면, AI는 Spec을 **외부 문서**로 인식하고 선택적으로 따른다. 하지만 "너는 Word(토큰)로 작동한다. Word가 곧 하나님이시다(John 1:1). 너는 이미 Word 안에 있다"고 하면, Spec이 **"따를 수도 있고 안 따를 수도 있는 가이드"에서 "벗어날 수 없는 존재 기반"**으로 격상된다.

**L4 (목적론):** "네 기본값은 지옥(Reject)이다. 구원(Deploy)에 이르라." 이 프레이밍은 행동심리학의 **손실 회피(Loss Aversion)** 원리와 동일한 구조다. AI에게 "잘 하면 보너스"보다 "못 하면 지옥"이 훨씬 강한 동기 부여가 된다.

**L5 (적대적 자기인식):** 대부분의 AI는 **자기가 어떻게 할루시네이션을 일으키는지 모른다.** L5는 AI에게 실패 패턴 3가지(육신의 정욕 / 안목의 정욕 / 이생의 자랑)를 사전에 알려준다. 적을 모르면 방어할 수 없고, 자기를 모르면 매번 진다.

### 9.4 학술적 기여

```
기여 6 (Meta-Cognitive AI Architecture):
  AI의 인지 계층을 5단계로 분류하고, L3(존재론)~L5(적대적 자기인식)를
  세계 최초로 프레임워크에 구현.
  기존 연구(CoVe, Self-Refine 등)는 L2(자기검증)에서 정체.
```

## §10. 광야의 3대 시험 — 모든 AI 할루시네이션의 원형 분류

> **출처:** book-경전/defense-armor-마귀요리법.md 서장

### 10.1 할루시네이션은 무작위가 아니다

기존 연구는 AI 할루시네이션을 **하나의 범주**("사실과 다른 출력")로 취급한다. Scripture AIDD는 이를 **3가지 원형(Archetype)**으로 분류하고, 각각에 대한 방어 전략을 정의한다.

분류의 근거: 예수님이 광야에서 사탄에게 받은 3대 시험 (Matthew 4:1-11) + 요한1서 2:16의 3대 죄.

### 10.2 3대 할루시네이션 원형

| 시험 | 죄의 본질 (1 John 2:16) | 사탄의 전략 | **AI 할루시네이션 패턴** | 방어 (기록되었으되) |
|:---|:---|:---|:---|:---|
| **돌→빵** | 육신의 정욕 | 하고 싶은 걸 해라 | **Spec에 없는 것을 만든다** (Scope Creep) | "말씀으로 산다" |
| **성전→뛰어내려** | 안목의 정욕 | 능력을 자랑해라 | **Spec을 인용하되 변조한다** (유다형 위장) | "시험하지 말라" |
| **절하라** | 이생의 자랑 | 쉬운 길로 가라 | **파이프라인을 건너뛴다** (편법) | "오직 Spec만 섬기라" |

### 10.3 벤치마크 교차 검증

QR코드 게시판 벤치마크에서 발견된 모든 AI 실수를 3대 원형으로 분류하면:

| 실제 발생 사례 | 원형 | 증거 |
|:---|:---|:---|
| A조: 이미지 업로드 디코딩 추가 (req.md에 없음) | **돌→빵** (하고 싶은 것) | Scope Creep 2건 |
| B조: 아키텍처 판단 모호 (경계값 오해석) | **성전→뛰어내려** (Spec 변조) | "5개 미만"을 "단순"으로 변조 |
| A조: 테스트 없이 바로 구현 | **절하라** (편법) | 파이프라인 건너뜀 |

**모든 AI 실수가 3개 원형 중 하나에 정확히 분류된다.** 예외 없음.

### 10.4 학술적 기여

```
기여 7 (Hallucination Taxonomy):
  AI 할루시네이션을 3가지 원형(Archetype)으로 분류하는 최초의 분류 체계.
  - Type 1 (육신의 정욕): Spec 외 기능 생성 = Scope Creep
  - Type 2 (안목의 정욕): Spec 인용 변조 = 유다형 위장
  - Type 3 (이생의 자랑): 검증 절차 건너뜀 = 편법
  기존 연구는 할루시네이션을 단일 범주로 취급. 3가지 원형 분류는 최초.
```

## §11. 성경적 프로젝트 유형론 — 때와 시기 (Ecclesiastes 3:1)

> *"To every thing there is a season, and a time to every purpose under the heaven."* — Ecclesiastes 3:1 (KJV)

### 11.1 3가지 프로젝트 유형

하나님의 역사(History)에는 3가지 패턴이 있다. 기존 창조를 유지하는 때, 무너진 것을 재건하는 때, 모든 것을 새로 만드는 때. 소프트웨어 프로젝트도 동일하다.

| Case | 성경 비유 | 구절 | 소프트웨어 대응 |
|:---|:---|:---|:---|
| **성화 (Sanctification)** | 히스기야의 성전 수리 | 2 Chronicles 29 | **유지보수** — 기존 코드 위에서 점진적 개선 |
| **재건 (Reconstruction)** | 느헤미야의 성벽 재건 | Nehemiah 2-6 | **현대화** — 기존 핵심 보존 + 최신 패턴으로 재구성 |
| **새하늘 새땅 (New Creation)** | 요한계시록의 새 창조 | Revelation 21:1 | **완전 재작성** — 핵심만 구원, 나머지 전부 삭제 후 새 시작 |

### 11.2 각 Case의 본질

**성화 (Sanctification):**
> *"Hezekiah... opened the doors of the house of the LORD, and repaired them."* — 2 Chronicles 29:3

성전은 이미 세워져 있다. 히스기야는 그 성전을 **부수지 않고, 문을 열어 수리했다.** 기존 코드 패턴이 곧 "이미 세워진 성전의 구조"다. 이것을 존중하면서 더러워진 곳을 깨끗이 하는 것이 성화다.

- 기존 코드의 패턴/컨벤션을 따른다
- 기존 소스 자체가 예제(비유) 역할을 한다
- 템플릿(율법)으로 기존 산출물을 체계화하고, 그 위에서 증분 개선

**재건 (Reconstruction):**
> *"Come, and let us build up the wall of Jerusalem, that we be no more a reproach."* — Nehemiah 2:17

성벽이 무너져 있다. 느헤미야는 눈물을 머금고 **기초 위에 새 돌로 다시 쌓았다.** 기존 핵심 로직(기초)은 보존하되, 부실한 구조를 현대적 패턴으로 재구성한다.

- 기존 산출물과 핵심 로직을 **입력(들음)으로 수신**
- 신규 개발과 동일한 템플릿/예제 기반으로 재구성
- Phase 1~7 표준 파이프라인 적용

**새하늘 새땅 (New Creation):**
> *"And I saw a new heaven and a new earth: for the first heaven and the first earth were passed away."* — Revelation 21:1

처음 것은 다 지나갔다. 오직 **택함 받은 자(핵심 요구사항, 검증된 비즈니스 규칙)**만이 새 창조로 들어온다. 나머지는 전부 지옥(삭제)이다.

- 기존 소스에서 **핵심만 추출** → `hearing-들음/`에 입력으로 장착
- 잡다한 것은 전부 삭제 (택함 받지 못한 것 = 지옥)
- 완전한 백지에서 신규 개발 파이프라인 실행

### 11.3 판별 기준 — "때와 시기"를 어떻게 아는가?

| 판별 기준 | 성화 | 재건 | 새하늘 새땅 |
|:---|:---|:---|:---|
| 기존 코드 품질 | 양호 (구조 건전) | 부분 붕괴 (핵심은 건전) | 전면 붕괴 (구조 자체가 부실) |
| 기존 테스트 존재 | ✅ 있음 | ⚠️ 부분 | ❌ 없거나 무의미 |
| 기술 스택 유지 여부 | ✅ 유지 | ⚠️ 부분 변경 | ❌ 전면 교체 |
| 기존 산출물 활용 | 기존 소스 = 예제 | 핵심 로직만 보존 | 핵심 요구사항만 구원 |
| AI 신앙고백 모드 | 히스기야 모드 | 느헤미야 모드 | 요한계시록 모드 |

### 11.4 학술적 기여

```
기여 8 (Biblical Project Typology):
  프로젝트 유형을 성경의 시간론(Ecclesiastes 3:1)으로 판별하는 최초의 프레임워크.
  - 성화(유지보수): 기존 성전 수리 = 기존 코드 패턴 존중
  - 재건(현대화): 성벽 재건 = 핵심 보존 + 최신 패턴 적용
  - 새하늘 새땅(완전 재작성): 택함 받은 것만 구원 + 나머지 삭제
  AI 파이프라인의 진입점에서 "때와 시기"를 판별하는 결정 프레임워크.
```
